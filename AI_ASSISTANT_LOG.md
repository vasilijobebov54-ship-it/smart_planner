# AI Assistant Log

История ключевых промптов к ИИ-ассистенту (Claude) при разработке проекта и решений, принятых на их основе.

## Этап 1. Проектирование

**Промпт:**
> Создай детальный план проекта (структуру папок и файлов) для приложения «Умный планировщик задач» на FastAPI + SQLite + Anthropic Claude. Включи модели для задач и пользователей, роуты CRUD, отдельный сервис для работы с ИИ и конфигурацию через переменные окружения. Поясни роль каждого файла одним предложением.

**Что я взял из ответа:**
- Разделение на `models.py` / `schemas.py` / `crud.py` / `ai_service.py` / `main.py` — стандартный паттерн для FastAPI, читаемый и тестируемый.
- Отдельный `database.py` с зависимостью `get_db()` — это позволяет в тестах подменить БД через `app.dependency_overrides`.
- Конфигурацию AI вынес в переменные окружения (`ANTHROPIC_API_KEY`, `ANTHROPIC_MODEL`).

**Что я отверг:**
- Предложение сразу добавить Alembic для миграций. Для прототипа на SQLite это overkill — `Base.metadata.create_all` достаточно.

## Этап 2. AI-сервис

**Промпт 1 (категоризация):**
> Напиши функцию `categorize_task(description)` на Python. Она должна вызывать Anthropic Messages API (`/v1/messages`) и просить модель отнести задачу к одной из категорий: работа, личное, здоровье, обучение, другое. Промпт должен явно требовать «верни только название категории одним словом». Используй httpx. Обработай таймауты и ошибки сети — функция не должна падать.

**Что было важно проконтролировать:**
- ИИ изначально предложил парсить ответ как `response["content"][0]["text"]`, но это сломается, если модель вернёт несколько content-блоков. Я переписал на цикл с проверкой `type == "text"`.
- Модель иногда отвечает с кавычками, точкой или фразой целиком («Эта задача — обучение»). Поэтому я **вынес парсинг в отдельную функцию `parse_category`**, которая ищет валидную категорию в ответе. Так её можно тестировать без сети.

**Промпт 2 (оценка времени):**
> Промпт для модели: «Оцени, сколько минут займёт эта задача. Верни только число». Напиши парсер, который достаёт число из любого ответа и ограничивает результат разумным диапазоном (1 минута – 24 часа).

**Решение:** регэксп `\d+` + `max(1, min(value, 1440))`. Защита от ситуаций «модель вернула 99999» или «модель ничего не вернула».

## Этап 3. Тесты и документация

**Промпт:**
> Напиши pytest-тесты для парсеров `parse_category`, `parse_minutes`, `parse_subtasks`. Не вызывай сеть. Также напиши тест роута POST /tasks через `fastapi.testclient.TestClient` с замокированным AI-сервисом через `monkeypatch`.

**Что взял:**
- Подмена `get_db` через `app.dependency_overrides` + отдельная SQLite-БД для тестов.
- `monkeypatch.setattr(ai_service, "categorize_task", ...)` — не дёргаем API в тестах.
- Фикстура `reset_db` с `autouse=True` — каждый тест с чистой БД.

**Документация:** OpenAPI/Swagger FastAPI генерирует автоматически из docstring и Pydantic-схем — отдельно его писать не нужно. Доступен по `/docs`.

## Этап 4. Code review от ИИ

**Промпт:**
> Проведи рефакторинг моего `ai_service.py`. Особое внимание удели обработке ошибок при вызове внешнего API и тестируемости.

**Топ-3 предложения, которые я применил:**

1. **Вынести парсеры из бизнес-логики.** Изначально парсинг был внутри `categorize_task`. ИИ предложил вынести `parse_category`/`parse_minutes`/`parse_subtasks` отдельными чистыми функциями. *Польза:* их теперь можно покрыть тестами без сети, и при смене модели (Ollama, локальная) не придётся переписывать парсинг.

2. **Безопасные дефолты вместо исключений.** Изначально при ошибке API сервис кидал HTTP 500. ИИ предложил возвращать `None` из `_call_claude` и фолбэки `"другое"` / `30` из парсеров. *Польза:* пользователь всё равно создаст задачу, даже если API временно недоступен. Деградация мягкая.

3. **Ограничение `max_tokens` и таймаут.** ИИ заметил, что в моём первом варианте таймаута не было и `max_tokens` стоял огромный. Это риск зависания и лишних трат. Поставил `timeout=20.0` и `max_tokens=20–300` в зависимости от задачи.

**Что отверг:**
- Предложение добавить retry с экспоненциальной задержкой. Для прототипа избыточно — если API временно лёг, лучше быстро вернуть дефолт, чем заставлять пользователя ждать.

## Этап 5. AI-агент для сложных задач

**Промпт (цепочка):**
> Напиши функцию `split_into_subtasks(description)`. Промпт к модели должен требовать строго формат:
> ```
> 1. ...
> 2. ...
> 3. ...
> ```
> Парсер должен извлекать ровно 3 подзадачи. Если модель ответила криво — дополнить дефолтными ("Спланировать", "Выполнить", "Проверить") до 3 штук.

**Реализация в `main.py`:** при создании задачи с `complexity=high` роут вызывает агента, получает 3 названия и создаёт 3 дочерние записи в БД (с `parent_id`). Каждая подзадача тоже проходит через `categorize_task` и `estimate_minutes`. Это **цепочка промптов**: один промпт разбивает, потом для каждого результата идёт ещё два промпта (категория + время).

## Что бы я улучшил при наличии времени

- Добавить аутентификацию (сейчас `owner_id` передаётся как query-параметр — для прототипа ок, для прода — JWT).
- Кешировать ответы ИИ для одинаковых описаний — экономия токенов.
- Batch-промпт: за один вызов получать и категорию, и минуты, и подзадачи — три вызова на одну задачу можно слепить в один.
- Перейти на async-клиент `httpx.AsyncClient` и сделать роуты `async` — FastAPI это умеет, а ИИ-вызовы I/O-bound.

## Главный вывод

ИИ-ассистент полезен на всех этапах, но **с обязательной проверкой**. Самые типичные мои правки после ИИ:
1. Перевернуть «бросать исключение» на «вернуть безопасный дефолт».
2. Вынести парсинг ответа модели в чистые функции.
3. Не доверять формату ответа модели — всегда писать защитный парсер с регэкспом и дефолтом.
